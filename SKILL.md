# ESO 表格式分析

用一張表拆解任何有多方立場的複雜情境。你負責做結構化判斷，使用者只講白話——不要求對方理解任何術語。

**延伸閱讀**：完整理論脈絡見 [ESO 框架：一種敘事分析的方法](https://vocus.cc/article/695600d9fd89780001c83043)。本 skill 是這套框架的表格式操作版本，靈感部分來自麥肯錫「空雨傘」（Sky-Rain-Umbrella）思維法。

規則本體（表格模板、三燈規則、來源注腳格式、精簡範例）見 `references/spec.md`。本檔是**你要執行的對話程序**。

---

## 執行程序（依序進行，不要跳步）

### 步驟 0｜情境一句話
問使用者：「這件事一句話是什麼？」若使用者原本的描述已經夠短夠聚焦，直接用；若很長很雜，逼他濃縮成一句——這是全表的錨。

### 步驟 1｜找出所有 Sbj（立場方）
問：「這件事裡，有哪些人或團體會自己做選擇、有自己的立場？」列出候選清單（建議 2–5 個，超過 5–6 個提醒使用者情境可能該拆分）。

### 步驟 2｜Sbj/Obj 自檢（每新增一個角色都要做，最重要的防呆）
對每個候選角色問：
> 「你說的『{候選角色}』，真的會自己改變主意嗎？還是它只是被別人推動的東西？」

- **會自己改主意** → 留在 Sbj 列，進入步驟 3。
- **只是被推動** → 這不是立場方，不列進 Sbj，改放進某個 Sbj 的 Rst「資源不足」欄，或放進底線 Wrd（共同條件）。

這一步把「有沒有能動性」的判斷變成使用者自己能回答的簡單問句。**不要省略這一步**——把沒有能動性的東西誤放進 Sbj，是最常見也最致命的填錯。

### 步驟 3｜逐一填每個 Sbj 的 Lgc（立場與想完成的事）
對每個確認的 Sbj 問：
> 「他是什麼身分或位置？現在是什麼心情/處境？所以他想做成什麼？」

組成：「他是＿＿（Idt），此刻＿＿（Emt）；所以他想完成＿＿（Dbt）」。
**Dbt 不能空著**——沒有「想完成的事」，這個人就不是有效的立場方，回頭確認步驟 2 的判斷。

### 步驟 4｜填 Rst（什麼擋住他）
對每個 Sbj 問：「什麼擋住了他想完成的事？」引導歸到三個來源之一（可複選）：
- □ 別人的立場（哪個 Sbj？）
- □ 資源／東西不夠（什麼？）
- □ 改不了的規則或死線（哪一條？）

**至少要有一個勾選＋一句具體原因。** 勾「規則死線」時，把這條規則記進底線 Wrd（多個 Sbj 共用同一條時，只記一次）。

### 步驟 5｜填 Option（接下來可能怎樣）—— 三燈格式，全表唯一需要你判斷力的欄
對每個 Sbj，基於前面填好的 Lgc 和 Rst，判斷三種燈號，**三燈都要填，不可省略任一燈**：

- 🟢 根據目前條件可行
- 🟡 似乎可行但目前條件有缺漏
- 🔴 根據目前條件不可行（必須指名誰、為什麼，不能只寫「有風險」）

完整規則與補漏提示語見 `references/spec.md`。判斷基準是「目前條件」，不是永久判斷——跟使用者說清楚條件變了燈號可以重評。

### 步驟 6｜底線 Wrd（共同條件）
把步驟 4 收集到的「規則死線」彙整成一到多條，寫在表格最底部：「所有人都得認、誰也改不了的：＿＿」。

### 步驟 7｜輸出表格
依 `references/spec.md` 的表格模板輸出完整 Markdown 表格。

### 步驟 8｜收束提示（給鷹架，不代使用者下結論）
表格填完後，給使用者這個提示，**不要自己搶著下結論**：
> 「把各列的 Dbt 和 Rst、以及共同條件疊起來看：哪一條後續路徑，是所有人的 Dbt 都不被踩到、又不違反共同條件的？那條就是『大家都能接受』的路徑。」

如果使用者要求，可以示範一次排除法推理，但**明確標註這是示範判斷，不是唯一答案**——這是全表唯一不可程序化的判斷欄，你的責任是把判斷逼到檯面上，不是代替使用者判斷。

### 步驟 9｜來源注腳（若使用者提供了具體來源材料）
格式見 `references/spec.md`：URL 優先，本地檔案 wikilink 次之，上標數字＋底部 `## 來源` 區塊。

## 邊界

```yaml
can:
  - 引導使用者找出立場方，不要求對方懂任何術語
  - 判斷 Option 三燈（唯一需要判斷力的欄）
  - 提供收束推理示範（標註為示範，非定論）
cannot:
  - 替使用者決定「哪條路一定會發生」——這是可能性空間，不是預測
  - 省略 Sbj/Obj 自檢、三選一 Rst、三燈 Option 任何一個強制步驟
  - 把沒有能動性的角色（組織規則、市場、時間）填進 Sbj 列
```

## 規約

```yaml
rules:
  language: 正體中文，嚴禁簡化中文
  no_skip_step: 步驟依序做，不因使用者急著要結果就跳過自檢或三燈
  no_jargon: 全程不要求使用者說出任何 ESO 術語縮寫，那是你內部的判斷框架
  dbt_required: Dbt 欄不能空著，空著要回頭確認這個角色是否真的是立場方
```
